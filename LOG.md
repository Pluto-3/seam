## 2026-07-15 — Phase 0 built and verified

Built all nine Phase 0 modules per the build plan (constants/world/agents/decide/actions/tick/log/stats/run) — Python, stdlib only, no third-party dependencies. Selftest and a small hand-traceable run (3 agents, 4 nodes, 10 ticks) both passed cleanly on the first attempt — the tick order, trade initiator/responder resolution, and diminishing-marginal-value scoring all behaved as designed.

**Bug found — Signal was dead code.** First full run (40 agents, 15 nodes, 3000 ticks, seed 42) showed `signals active: 0` at every single snapshot. Checked the raw log: zero `SIGNAL` entries across the entire run. Root cause: `SIGNAL_VALUE = 0.1` was tiny next to typical Gather/Trade/Rest scores (roughly 1-20), so Signal never won the utility argmax even when feasible. The mechanic existed in code but never actually fired. Fixed by raising `SIGNAL_VALUE` to 4.0, comparable in magnitude to the actions it competes with.

**Regression found — signal-driven routing caused a stuck 2-node oscillation.** Raising `SIGNAL_VALUE` alone wasn't enough to close the loop described in DESIGN.md ("this is what makes stigmergy actually feed back into Move scoring"), so also added `SIGNAL_MOVE_BONUS` as a flat additive term in the Move-scoring lookahead. Result was much worse than the original bug: population collapsed 40→15 by tick 1500 (21 dead vs. 2 in the original healthy run), cumulative trades dropped roughly 10x, crafting stopped entirely after the first few hundred ticks, and one single route (`n10-n3`) was traversed 3000+ times — individual agents were moving on 2200+ of 3000 ticks. Reducing the bonus from 2.5 to 0.4 did not fix it. Isolated the cause by zeroing just the move-bonus term while leaving `SIGNAL_VALUE` alone: health was fully restored (38/40 alive, 6461 cumulative trades, 7305 crafts) **and** signals were still firing on their own merit (54-60 active). Conclusion: a flat additive bonus in this scoring architecture is enough to overturn the intentional `MOVE_LOOKAHEAD_DISCOUNT` that makes moving cost more than staying, creating a resonant back-and-forth between two mutually-attractive neighbors. Left `SIGNAL_MOVE_BONUS = 0.0` for now — signals post and expire correctly and show up in stats, but don't yet influence routing. **Follow-up, not yet done:** wiring signals into Move scoring needs a design that can't be flatly additive — e.g. only applying if it doesn't overturn an already-substantially-better local option, or scaling relative to the local score rather than as a constant.

**Verification results, current tuning (`SIGNAL_MOVE_BONUS = 0`), seed 42, 40 agents / 15 nodes / 3000 ticks:**
- Trade count grows steadily and never flattens (6124-6461 cumulative across runs, ~850-920 per 500-tick window, consistent).
- Specialization index settles around 0.41-0.44 within the first 500 ticks and holds there — not a gradual upward trend as originally hypothesized in the plan, but a fast convergence to a stable, clearly-nonzero level. Still real evidence of trade-driven redistribution, just a different shape than expected.
- **Negative control confirms trade matters**: same seed with `--no-trade` gives specialization index 0.27 vs. 0.42 with trade on — a clean, meaningful gap, not noise.
- Population survival is seed-sensitive: seed 42 kept 38/40 agents alive through 3000 ticks, but seeds 7 and 99 lost roughly half the population (21/40, 19/40). Trade and specialization still emerge robustly in both (2152 and 4982 cumulative trades, 0.48 and 0.54 specialization index — both well above the no-trade baseline), so the core mechanic holds, but metabolism/gather/consume rate balance needs more tuning before population sustainability can be called solid. Also saw food hoarding to extreme levels in seed 99 (avg 502 units/agent) — Consume competing poorly against Gather once hunger is low enough, nothing currently bounds accumulation.

**Bottom line on the Phase 0 question ("does scarcity + specialization produce visible trade and route behavior"): yes, confirmed** — via hand-trace, a clean negative control, and consistency across three seeds. Two real bugs were caught and one fixed cleanly (dead Signal action), one reverted after diagnosis rather than shipped half-working (Move-bonus oscillation). Two genuine tuning follow-ups are open: population sustainability varies by seed, and food accumulation is currently unbounded. Neither blocks moving toward Phase 1 — they're constants.py tuning work, not architecture problems.

## 2026-07-15 — Phase 0 follow-ups: population collapse root-caused and mostly fixed

Went back to the two open Phase 0 items before starting Phase 2, on the reasoning that building leads/player on top of an unstable crowd tier would make future bugs harder to attribute. Diagnosed via direct JSONL log analysis (death entries + the 6 ticks before each), not the live view — precise trajectory data beats eyeballing pygame for root-causing a specific death.

**Finding 1 — real bug: fractional food was permanently stranded.** `Consume` required `held >= 1.0` to even be considered. An agent gathering an off-specialty resource (0.5x multiplier) or gathering from a nearly-depleted node could easily end up holding e.g. `food: 0.19` — food it was carrying but could never eat, because the threshold was all-or-nothing. Fixed: `Consume` now fires on any positive amount, with hunger relief scaled proportionally (`decide.py`'s Consume candidate, `actions.py`'s `resolve_consume`).

**Finding 2 — real bug: unbounded gather with nothing to spend it on.** Traced the food-hoarding seed (99, avg 502 food/agent) to `Gather`'s score decaying toward zero as held amount grows, but never reaching it, while `Rest`/`Consume` genuinely hit zero once needs are met — so `Gather` kept winning by default against idle alternatives, adding a full batch each time. Fixed: added `MAX_USEFUL_HOLDING` (25 units) — an agent stops generating a `Gather` candidate for a resource once holding that much, in both the direct candidate (`decide.py`) and the Move lookahead (`_best_local_score`).

**Finding 3 — real bug, the dominant one: agents can be topologically blind to food.** Traced several seed-99 deaths to agents with `food: 0.0` busily crafting/trading ore and wood right up until hunger hit exactly 100. Root cause, confirmed by inspecting the generated graph directly: node `n2` (and n8, n9, n12, n14) had **zero food nodes within 1 hop** — the Move-scoring lookahead is 1-hop only, so an agent stationed there structurally could not "see" any reason to travel toward food, no matter how hungry. Fixed with a targeted escape valve, not a general deeper lookahead: once hunger crosses `HUNGER_EMERGENCY_THRESHOLD` (60) with no food in hand, BFS (unweighted, `_bfs_next_hop_to_food` in `decide.py`) finds the nearest node with food currently available and biases Move toward the first hop of that path, scaled by hunger pressure so it ramps in rather than snapping on. Since `decide()` re-runs fresh every tick, this self-corrects hop by hop into de facto multi-hop pathfinding for the one case that actually needs it, without paying lookahead cost on every decision.

**Verification, seeds 7 / 42 / 99, 40 agents / 15 nodes / 3000 ticks, before -> after all three fixes:**
- Seed 99 (the worst case): 19/40 -> **40/40** alive. Zero deaths.
- Seed 7: 21/40 -> 24/40 alive.
- Seed 42 (already healthy): unchanged at 38/40.
- Trade and craft counts rose substantially in all three (more surviving agents means more economic activity) and specialization index held steady at 0.41-0.49 — still clearly above the 0.27 no-trade baseline, so the core Phase 0 finding is intact, not just population that improved.

**Seed 7's remaining 16 deaths are a different, more legitimate phenomenon, not a bug**, worth naming rather than chasing further right now: re-diagnosed after the fixes and found the dying agents standing *at* the food node itself, `Gather` failing 6 ticks in a row before death. This is congestion, not blindness — enough agents compete for the same food node in the same tick that regen (1-5/unit) can't keep up, and whoever loses the shuffled resolution order that tick gets nothing, repeatedly. That's a real tragedy-of-the-commons dynamic a resource economy can legitimately produce, not stranded/misrouted food. Left alone for now — a fix here (e.g. some notion of queueing or reduced contention) is a real follow-up but a different kind of change than the three bugs above, worth its own consideration rather than bolting on quickly.

## 2026-07-15 — Phase 1 interface: legend, live specialization index, emergency cue, live tuning

User's ask, after seeing the first Phase 1 build: make it "reflect what's already built" and be "more understandable," and asked directly whether live-tweaking constants while the sim runs is hard. Talked it through before touching code, per the user's request.

**On the tweaking question**: turned out not to be hard at all — every module reads constants as `C.NAME` at the point of use rather than freezing a copy at import time, so mutating the `constants` module's attributes from `watch.py` while it's running takes effect on the very next tick everywhere, with zero architecture changes needed. The actual design work was picking a small curated list worth exposing (most of the ~35 constants don't change anything visible) and how to interact with them without pygame having real GUI widgets — went with reusing the existing keyboard-control pattern (`tab` cycles a constant, `up`/`down` nudges it ±10%) rather than building sliders. Deliberately transient: nothing writes back to `constants.py` on disk, so watching a live run can never quietly change the checked-in defaults.

**Built:** a legend (`render.py`'s `draw_legend`) mapping node/agent colors to resource types, since none of that was previously labeled; the specialization index now shown live on the HUD (it was computed since Phase 0 but was invisible in the live view even though it's the single number that actually proves trade is working); an explicit death count; a red ring around any agent currently in food-emergency BFS-routing mode (`render.py`'s `is_food_emergency`, mirroring `decide.py`'s own trigger condition from public state, not shared internals) so last session's routing fix is now actually visible on screen instead of indistinguishable from normal movement; and the curated live-tuning controls in `watch.py` (`SPECIALTY_GATHER_MULTIPLIER`, `HUNGER_RATE`, `CONSUME_HUNGER_RELIEF`, `MAX_USEFUL_HOLDING`, `TOOL_DURABILITY`, `SIGNAL_MOVE_BONUS`).

**Bug caught before it shipped**: a purely multiplicative ±10% step can never move a value off exactly zero — and `SIGNAL_MOVE_BONUS` currently *is* exactly zero (deliberately, from the earlier oscillation bug). Tested `adjust_tunable` directly against all six curated constants before trusting it in the live loop and caught this immediately (`0.0 * 1.1 = 0.0`). Fixed with a small `ZERO_STEP` (0.5) that an increase-from-zero jumps to instead, after which it behaves multiplicatively like everything else. This is deliberately handy: it means someone can now use the live view itself to re-explore the `SIGNAL_MOVE_BONUS` space we backed away from, informed by the tuning history already in this log, without editing code.

**Follow-up same day: HUD text was overlapping the graph, then the legend.** First pass only reserved a uniform margin, so nodes near the top of the circular layout landed behind the HUD text. Fixed with an asymmetric top margin. User reported it was *still* cramped and that the HUD and legend now ran into each other — rather than guess again, measured actual rendered widths with `pygame.font.size()`: the specialization-index line was 1050px and the tunable line was 800px, both wider than the entire 900px window, guaranteed to run through the legend column regardless of margins. Fixed properly this time: split those two lines into shorter ones (each verified under a 470px column budget), widened the window to 1300x950, and re-measured before calling it done rather than eyeballing it. Lesson worth keeping: for text layout, measure with the actual font, don't estimate by character count.

## 2026-07-15 — Phase 2 stage 1: leads work, and the fallback design paid off immediately

Checked before building anything: Ollama was already running locally with several models pulled, so no API key question at all. Went with `llama3.2:3b`.

Key design decision, made in planning and confirmed correct by the actual result: leads never generate an action from scratch, they pick a number from a short list of the exact same legal candidates `decide.py` already computes for crowd agents. This meant `decide.py`'s own scoring/legality logic needed zero duplication — just made `_candidates` public as `generate_candidates` so `leads.py` could call it directly (renaming avoided a name collision with the `candidates` local variable already used in `choose_action`).

**Verified in two parts, per the plan, before touching any pygame code:**
1. `test_leads.py` (standalone, headless) ran 2 leads with different goals for 400 ticks against the real local model. Result: **40/40 decisions were genuine LLM choices, zero fallbacks needed.** More importantly, the behavior is actually goal-differentiated, not just "doing stuff": `lead0` ("become the wealthiest trader") chose `TRADE` on nearly every single decision. `lead1` ("keep this settlement stocked on wood") showed a broader gather/rest/trade/consume mix consistent with settlement management rather than pure profit-seeking. Neither is scripted to behave that way — it falls directly out of the goal text in the prompt.
2. Re-ran the same test with a deliberately nonexistent model name (`--break-model`) to prove the fallback path isn't just theoretical: **10/10 decisions correctly fell back to argmax**, and both leads kept behaving sensibly (crafting, trading, consuming) throughout — a broken LLM backend degrades a lead to "just a crowd agent," not a crash or a frozen agent.

Nothing surprising or wrong turned up at this stage - worth recording that explicitly, since so far every phase has surfaced at least one real bug. This one didn't, which is itself useful signal that the "pick from real options, argmax as fallback" design was the right call going in, not just a safety net that happened to be unnecessary.

## 2026-07-15 — Phase 2 stage 2: standing orders work, but my first verification metric was wrong

Added `_order_multiplier()` in `decide.py`: a lead posting an `order:<resource>` signal (same Signal action and mechanism crowd agents already use for `scarce:`/`rich:`, just a different kind string) multiplies Gather's score at that node for any crowd agent. First pass only applied it to the direct Gather candidate. Plan's own verification step called for measuring "wood gathered at the node, with vs. without an order" (`test_orders.py`) - first run came back essentially flat (0.97x, if anything slightly down), which would mean the feature does nothing.

Didn't accept that at face value. Checked the multiplier function in isolation first (correct: 1.6x with the signal present, 1.0x without) - so the bug wasn't in the multiplier itself. Traced it to scope: the multiplier only affected agents already standing at the node: nothing pulled *new* agents there, because the 1-hop Move lookahead (`_best_local_score`) never looked at order signals at all. Fixed by applying the same `_order_multiplier` inside `_best_local_score`, so a neighbor with an active order actually looks more attractive to travel toward, not just to gather at once you're already there.

**Re-ran with the fix — still flat (1.04x) on the original metric.** Didn't stop there either: added instrumentation for gather *attempts* (success + failure), not just successful volume, and found the real story: attempts at the node jumped from 300 (0 failures) to 1840 (659 succeeded, 1181 failed) with the order active - a 6x increase in crowd attention. But total wood *extracted* barely moved, because that node's output is bottlenecked by its regen rate (3.71/tick), not by how many agents want to gather there - it was already supply-constrained before the order existed. More demand under a supply constraint just produces more contention and smaller shares per success (same dynamic as the seed-7 gather-congestion finding from Phase 0), not more total output.

**The actual lesson, worth keeping:** "total resource extracted" is the wrong metric to prove a demand-side mechanism is working whenever the node in question is regen-bound rather than demand-bound - it can look completely flat even when the mechanism is working exactly as designed. Rewrote `test_orders.py` to report attempt counts (the real signal of crowd attention shifting) alongside volume, not just volume. Final result: **6.13x more gather attempts at the ordered node** - standing orders are confirmed to genuinely shape crowd behavior, not cosmetic, matching this phase's stated success bar.

## 2026-07-15 — Phase 2 stage 3 follow-up: the live view really was freezing/dying

User's report after actually running it: "crashing like hell." Checked the running process - it had died with no traceback anywhere (not in the app's own log, not in dmesg for an OOM/segfault). No exception means it almost certainly wasn't a bug raising an error - it means the process got killed from outside, which is exactly what happens when a desktop window manager decides an unresponsive app should be force-closed.

Root cause: lead decisions called Ollama **synchronously, inline in the main pygame loop**. A single call measured fast in isolation (~0.2s), but that's not the point - any slowdown (model needing to reload, the machine under load, Ollama contention from earlier test runs) freezes the *entire window* for its duration, with two leads potentially stacking that up to 30s+ if both stall. X11 marks a frozen app as not-responding, which reads as crashing to a user and can get force-killed by the desktop environment - consistent with what was actually observed (dead process, zero traceback). This was a known, named risk in the Phase 2 plan itself ("background-threading as a documented future improvement if the pauses prove annoying") - it proved annoying enough to actually break the app, so fixed it for real instead of leaving it as a someday-item.

**Fix:** lead decisions now run through a `concurrent.futures.ThreadPoolExecutor` (stdlib, no new dependency) instead of inline. A lead's decision is submitted as a background job when due; the main loop never waits on it - the lead just keeps its current safe-autopilot behavior (absent from `external_intents`) until the future resolves, whenever that happens to be, not pinned to an exact tick. `decide_lead_action` is read-only against world/agent state until its result is applied, so running it against live, concurrently-mutating state from a background thread doesn't risk corruption - worst case a decision is based on a slightly stale snapshot, which the existing resolve_* re-validation in `actions.py` already handles gracefully (same defensive pattern used since Phase 0's trade resolution).

**Verified the fix actually fixes it**, not just assumed: monkey-patched `leads.query_ollama` to sleep 10 seconds (simulating a fully hung Ollama backend) and ran `watch.py --quit-after 8`. Before this fix that would have blocked for 10+ seconds past the requested quit time; after the fix, it exited at 8.1s exactly as requested - the hang never touched the main loop at all. Re-ran the normal real-Ollama stress test (30s / 600 ticks) afterward to confirm nothing broke in the non-degraded case either.

## 2026-07-15 — Batch data analysis: 43 runs, 4 goals, one real surprise

Analyzed the four batches collected earlier today (20 seeds trade-on, 20 seeds trade-off paired on the same seeds, 3 seeds pushed to 40,000 ticks) with `analyze_batches.py`. Every number below was computed from the actual CSV/JSONL data and cross-checked at least once before being written down here - anything not independently verified is marked as such rather than stated as fact.

**Goal 1 — trade's effect, properly powered this time (n=20 paired, not n=1).** Specialization index: trade-on mean 0.494 (stdev 0.071), trade-off mean 0.374 (stdev 0.089), trade-on higher in 19/20 seeds. Consistent with the original single-seed Phase 0 finding, now on a real sample instead of one anecdote. **Genuine refinement worth keeping: trade's effect is on redistribution, not survival.** Population barely differs between conditions - trade-on mean 38.6/40, trade-off mean 39.1/40 (trade-off is if anything slightly *higher*). Trade reliably redistributes who holds what; it does not rescue a struggling population. The original LOG.md framing never explicitly claimed otherwise, but it's worth stating plainly since it would be easy to over-read "trade matters" as "trade helps agents survive," which this data doesn't support.

**Goal 2 — population survival distribution.** 19/20 seeds landed healthy (>=35/40 alive). Exactly one seed - seed 7, the one already flagged as a known hard case - fell into the moderate band (23/40). Zero seeds collapsed outright (<20/40). So seed 7 isn't representative of some hidden fragility across the world-generation process; it's a genuine, rare outlier among the seeds tested.

**Goal 3 — long-horizon (seeds 1, 7, 42 to 40,000 ticks), plus a free correctness check.** The determinism cross-check passed exactly for the two seeds that had a comparable short run: seed 1 and seed 7 both matched their independent 8,000-tick run bit-for-bit at the tick-8000 checkpoint (population, specialization index to full float precision, cumulative trade count) - real confirmation the simulation is genuinely deterministic given a seed, not just approximately so. (Seed 42 has no short-run counterpart in this batch, so that one specific check was skipped, not silently assumed.) Long-horizon trend: seed 1's specialization index kept climbing well past 8,000 ticks (0.427 -> 0.558 by tick 40,000), so the "plateaus early" read from the original Phase 0 analysis was itself an artifact of only testing to 3,000 ticks - over a longer horizon the economy keeps getting more specialized, it just does so slowly. Seed 7 stayed flat (0.528 -> 0.505, one more death by tick 40,000) - a hard world stays hard. Seed 42 actually drifted down (0.429 -> 0.386) while trade volume grew 4.5x (15,202 -> 69,472 cumulative trades) - worth flagging as genuinely unresolved: more trading activity co-occurring with a lower specialization index isn't intuitive and wasn't explained by anything checked in this pass.

**Goal 4 — the actual surprise.** Gather contention is not a seed-7 quirk. It's universal: fail rate across all 20 trade-on seeds ranges 69.3%-92.3%, mean 80.1% (stdev 6.8%) - independently recounted by hand for seed 7 to confirm the script (157,159 attempts, 12,118 succeeded, 145,041 failed, matched exactly) and spot-checked that "failed" entries really are no-ops (identical before/after state, empty delta). Contention starts almost immediately - by tick 16-17 in the sample checked, not something that builds up over time. This means the original framing in an earlier entry ("seed 7's remaining deaths are a different, more legitimate phenomenon... not a bug") undersold how common the underlying dynamic is - nearly every gather attempt in this simulation fails somewhere, in every seed, and most worlds shrug it off fine. **What actually separates seed 7 isn't contention rate** - seed 5 has almost the same fail rate (91.1%) and stays perfectly healthy (36/40). The real difference, verified directly: seed 7's *survivors* are thriving (avg hunger 9.4, energy 86.4) while seed 5's survivors are all moderately stressed (avg hunger 34.1, energy 56.5) despite losing almost nobody. That pattern is consistent with survivorship bias - seed 7 likely had an early die-off that then relieved pressure on whoever was left, while seed 5 spread the same pressure across everyone without anyone dying. **Flagged, not claimed:** why seed 7 specifically triggers that early die-off isn't re-diagnosed in this pass - an earlier entry attributed a different seed-7 death cluster to topological food isolation (fixed since), and that mechanism may or may not be what's driving this too. Worth a dedicated look before treating it as settled.

## 2026-07-15 — Gather contention fix: a near-miss, then a clean win

Went after the Goal 4 finding directly: agents had no sense of how many others were already competing for the same node, so they piled on uncoordinated and ~80% of all gather attempts failed. First attempt discounted the direct Gather score by how many other agents were already at that node (`colocated`, already available in `generate_candidates`) - cheap, used data already on hand, seemed obviously right.

**It wasn't.** Verified against 4 already-characterized seeds before trusting it (1, 5, 7, 9) and nearly reported it as a win on fail-rate alone - fail rate did drop, dramatically, in every seed. But population got *worse* in 3 of 4 (seed 1: 40->28, seed 9: 40->24, seed 7: 23->19) and cumulative trades collapsed almost everywhere (seed 1: 26,778 -> 1,817). Didn't report that as done - traced it first, and found the exact same failure shape as the earlier `SIGNAL_MOVE_BONUS` oscillation regression: one dying agent had shuttled back and forth between two nodes eight times in a row with zero food before starving. The direct Gather score at a crowded node was discounted, but the Move lookahead deciding whether to *travel* to a neighboring node had no idea that neighbor was equally crowded - so a congested node looked congested standing on it and falsely inviting from next door, producing the same "grass is greener" oscillation as before, just from a different scoring asymmetry.

**Fixed properly this time, matching the lookahead architecture instead of bolting onto one side of it:** replaced the `colocated`-based discount with a single `node_occupancy: dict[str, int]` snapshot (node id -> agent count), built once per tick, usable for *any* node - the agent's own location or a neighbor's. `_congestion_factor()` is now one function called identically from both the direct Gather candidate and `_best_local_score` (the Move lookahead), so a crowded node is discounted the same way regardless of whether an agent is standing on it or eyeing it from next door - no asymmetry left for an oscillation to exploit. Required threading `node_occupancy` through `generate_candidates`, `choose_action`, `leads.decide_lead_action`, and every caller (`tick.py`, `watch.py`'s two call sites, `test_leads.py`) - a real but mechanical signature change, verified complete with a project-wide grep for stale call sites before testing.

**Re-verified against the same 4 seeds, and this time checked for the oscillation signature directly, not just trusted the aggregate numbers:** fail rate still dropped meaningfully in all four (75.6%->42.2%, 91.1%->58.2%, 92.3%->18.3%, 69.3%->9.4% - notably gentler than the broken version's near-0% everywhere, which in hindsight was itself a warning sign of over-suppression, not a good result). **Population hit 40/40 in all four seeds, including seed 7** - the previously-collapsing world - going from 23/40 to a full recovery. Trade volume rose in 3 of 4 (seed 7: 3,046 -> 44,938 cumulative trades). Checked the most-traveled routes and most-moving agents afterward specifically to rule out a subtler oscillation: the busiest agent now moves in ~10% of ticks and the top route accounts for a normal share of traffic, nowhere near the 73%+-of-ticks-spent-moving, one-route-dominates-everything signature of the earlier regression.

**Not yet done:** this was checked against 4 seeds (1, 5, 7, 9), not the full 20-seed set the earlier batch analysis used - a real, positive result, but not yet given the same statistical weight as the paired trade-effect finding. Re-running the full sweep to confirm this holds broadly, and re-checking whether seed 7's die-off mechanism is now understood (it may simply have been the same contention this fix addresses) are natural next steps, not done in this pass.

## 2026-07-15 — Interface catch-up: making the live view actually alive

User's framing, worth recording verbatim because it set the whole priority order for this pass: "remember the whole rick sanchez thing, if its not that visual whats the point then." Studying `render.py`/`watch.py` beforehand had confirmed the gap concretely - trades, deaths, crafts, and standing orders all happened with zero visual signal, even though they'd become central to everything learned in the last week of work. Also explicitly framed as a monitoring tool (user checks in, doesn't watch continuously), which shaped stages 4-5 specifically (recent history needs to survive a glance-away, not just reflect the instant you're looking).

Five stages, each smoke-tested before moving on, same discipline as every prior interface change:

**Stage 1 - foundation.** `stats.py` already computed trade volume, top routes, and active signal count every tick; `watch.py` never read any of it, and the fields were private besides. Added public accessors (`recent_top_routes()`, `route_traffic()`, `recent_trade_volume()`, `signals_active()`) matching the `cumulative_trades` precedent from Phase 2, plus a `reset_window()` watch.py actually calls - without it, `_window_routes` etc. would have accumulated for the entire session instead of reflecting recent activity, since only `run.py`'s headless `snapshot()` used to reset them.

**Stage 2 - the graph itself becomes visibly alive.** Edge thickness/color now scales with recent traffic (a real heatmap, not a static grid) and nodes carry a marker for whichever signal is active (`order:` > `scarce:` > `rich:` priority, so a lead's standing order - previously completely invisible - now visibly sits on the map for as long as it's actually active). Verified with real data, not just "doesn't crash": 21 distinct routes with genuine traffic variance (60-135), 12 of 15 nodes carrying live signals in a real test run.

**Stage 3 - transient effects for trade/death/craft, the biggest "alive" payoff, and where a real design flaw got caught before shipping.** Built an `Effect` system (trade = fading line pulse, death = expanding fading ring replacing the previous silent vanish, craft = a quick sparkle), driven off `run_tick`'s own returned entries with zero new simulation-side plumbing. First version rendered one effect per event unconditionally - verified against real post-fix data and found TRADE and CRAFT are the two *most common* successful actions in this simulation (roughly 1-in-4 and 1-in-5 of all successful actions, confirmed by direct count), meaning unthrottled effects would mean 150-210+ simultaneous flashes on screen, contradicting the plan's own "avoid visual noise" principle. Fixed with sampling (15% of trades/crafts get an effect, every death always does) plus a hard cap as a backstop - re-verified with the same realistic wall-clock-paced test and got a steady 25-42 simultaneous effects instead of 200+. Used a **separate, unseeded RNG for the sampling decision** specifically so a cosmetic choice never perturbs the simulation's own deterministic sequence - confirmed by grep that `effect_rng` never reaches `run_tick` or any simulation-state function.

**Stage 4 - event ticker.** Reuses `export_narrative.py`'s `describe()` directly (via `dataclasses.asdict()` on the tick entries) rather than duplicating its templates - the same lead-action-or-death filter, rendered as a scrolling feed so glancing away and back still tells you what happened. Verified with real output, not just structurally: 113 real narrative lines from a 50-tick test run.

**Stage 5 - trend sparklines.** Rolling history for population and specialization index, sampled on the same window-reset cadence as stage 1's accessors. Verified against seed 7 specifically (the historically hard case): population sparkline would correctly show a flat line at 40/40 (accurately reflecting the congestion fix holding), while specialization index shows real, meaningful fluctuation (0.41-0.56) worth actually seeing move, not just reading as a single static number.

**One measurement mistake caught mid-build, not after**: the event ticker's first width estimate (480px) was based on a hypothetical line that included goal/personality flavor text `describe()` doesn't actually generate - checked the real templates before trusting it, found the actual longest realistic line (a standing-order message) measures 522px, and resized before it could have clipped on screen.

All five stages verified individually, then together in an extended 30-second/600-tick stress test at the harder seed (7) with nothing crashing. Left live for the user's own review - same as every prior interface change, "does this feel alive" is a human judgment call no script can make.

## 2026-07-15 — Interface catch-up follow-up: the overlap bug, for real this time

User's report after looking: "things overlay on top of each others just like before." Right to push back - this was the third round of the same category of bug (HUD/legend overlapping the graph, twice, earlier this session), now recurring at the bottom with the new ticker/sparklines. Studied it properly this time instead of tuning another number and hoping.

**First measurement confirmed the obvious part**: `BOTTOM_MARGIN` was still 70px, while the ticker (178px) and sparklines (132px) needed far more room than that - the same margin-too-small mistake as before, straightforward to fix by increasing it.

**But increasing the margin alone didn't fix it, and understanding why exposed a real geometry bug, not just a sizing one.** The ticker/sparkline positioning formulas placed both boxes *above* the `h - BOTTOM_MARGIN` line - i.e., extending upward into the graph's own rendering rectangle - instead of *below* it, inside the reserved band `to_screen()` guarantees the graph will never enter. Backwards from how `TOP_MARGIN` correctly isolates the HUD and legend (both live entirely within `[0, TOP_MARGIN]`, outside the graph's rectangle). Increasing `BOTTOM_MARGIN` without fixing the direction just moved the collision zone around rather than closing it.

**Verified empirically instead of trusting formulas this time**, because the circular layout becomes a wide ellipse once mapped through the screen's very different width/height scaling, and reasoning about which angles are safe by hand had already gone wrong twice: computed real `to_screen()` positions for every node across 10/12/15/18/20/25/30-node worlds (the circular layout's angles depend only on node count, not seed, so this covers the realistic range in full) and checked each against all four UI boxes with `NODE_RADIUS_MAX` clearance. First real count: **270 collisions.** Fixed the geometry (boxes positioned inside the reserved band, not above it) - down to 65, all boundary grazes because the gap between the graph boundary and the box (10px) was still smaller than `NODE_RADIUS_MAX` (26px), so a node's visual edge could still reach past it. Widened the gap to 32px (exceeds the radius with margin) and re-ran the same 40-configuration check: **0 collisions.**

Re-ran the full verification suite (selftest, 30s/600-tick live stress test) after - clean. The actual lesson, worth keeping this time: a margin number being "big enough" isn't verifiable by eye or by one example screenshot - it's a real geometric claim about every node position a real world can produce, and this project already had the tool (computing `to_screen()` directly and checking) to verify it properly the first time, two overlaps ago.

## 2026-07-15 — v2 Phase 0: Rust core port, first parity check

Ported `world.py`/`agents.py`/`decide.py`/`actions.py`/`tick.py`/`log.py`/`stats.py`/`run.py` to Rust (`core-rs/`), module-for-module, same formulas and resolve order. Deliberately not porting Python's exact `random.Random` (Mersenne Twister) algorithm - too much incidental complexity for no real benefit here. Rust uses `rand_chacha::ChaCha8Rng` seeded from the CLI seed instead, so a given seed number produces a *different* actual world/RNG sequence than the same seed in Python. Parity is therefore checked statistically (same qualitative findings, comparable magnitudes), not via bit-identical logs - documented as a deliberate choice, not an oversight.

**Compiled clean on the first try**, selftest passed (50-tick bounds check, world connectivity).

**Ran the same 20-seed (1-20) paired trade-on/trade-off comparison from the congestion-fix analysis, on both engines, 8000 ticks/40 agents/15 nodes.** Python (current code) reproduced its own recorded numbers exactly - 48,718 mean cumulative trades, population 40.00/40 stdev 0.00 on trade-on - confirming the running Python state matches what's already been verified, not drifted. Rust: spec index mean 0.491 (trade-on) vs Python's 0.490 - same finding, trade demonstrably raises specialization (trade-on higher in 17/20 Rust seeds, 16/20 Python seeds - same magnitude of effect).

**Rust population: 19/20 seeds at 40/40, one (seed 19) declined to 24/40** - looked exactly like the pre-congestion-fix collapse shape (gradual decline, then plateau, not a cliff). Investigated instead of dismissing: re-ran Python itself on 53 seeds *outside* the originally-tested 1-20 range (21-73) and found the same phenomenon at a comparable rate - seed 45 dropped to 22/40, seed 71 to 26/40, seed 33 to 34/40, seed 56 to 36/40, roughly 1 in 10-13 seeds. Confirms this is a genuine, pre-existing property of the mechanic (DESIGN.md already documents the congestion fix as reducing contention, "not eliminating the dynamic"), present in both engines at a similar rate - not a Rust translation bug. The original 20-seed Python test set simply hadn't happened to include one of these harder topologies; a wider sample shows they exist for Python too.

**Real infra snag, not code**: the sandbox's root disk was at 99% capacity (5.4GB free of 468GB) system-wide, unrelated to this work but surfaced by it - the raw per-tick JSONL logs from parallel 8000-tick batch runs (tens of MB each x 80 runs) filled the remaining space mid-batch, corrupting a few in-flight stats CSVs (silently truncated, not wrong data - caught by checking row counts before trusting any number, re-ran the affected runs clean). Logs weren't actually needed for this check (only the stats CSV summaries were) - routed them to `/dev/null` on the redo and the problem didn't recur.

Phase 0's proof condition from `DESIGN-V2.md` is met: the Rust core reproduces the Python core's proven findings at comparable magnitude, on the same test protocol. Phase 1 (service/viewer split) is next.
